Literate Programming

In his 1984 work on “Literate Programming,” Donald Knuth calls for “better documentation of programs, and that we can best achieve this by considering programs to be works of literature.” It doesn’t seem like it would be efficient to put in more work to do better documentation of programs during the time. But Knuth himself references that doing so would make the code “more easily understood and maintained.” I believe that as a professor of computer science, Knuth understood the value in making code readable for others and was seeing the future of coding as being one in which sharing code and communicating it would enable a stronger group of computer scientists than those who could not easily share and communicate their code to others. Even if the code is not shared, it may not be intelligible to a future version of oneself if it’s not set up that way to begin with.

As far as other perceived benefits of literate programming, I believe it is a good way to be more thoughtful about code and possibly prevent errors through this thoughtfulness.

Additionally, I believe literate programming is a skill that will hone the right characteristics into shape. If you can put yourself in someone else’s shoes to create code that is user-friendly, it will only sharpen these skills and make it easier to create useful and relevant products for others. It’ll also make it easier to collaborate and work with others when you exercise the ability to look at something through another’s eyes.

While literate programming may not be the norm, it is even more relevant today as it was in Knuth’s text in 1984 when he called for more human-friendly coding. Code is now so prevalent to our lives and society that it would seem a shame not to create it in a way that is reproducible and able to be shared among users who could improve upon it.
